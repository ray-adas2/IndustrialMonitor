## 四、核心技术点地图

### 4.1 生产者-消费者模式（阶段一/二）

```
┌─────────────────────────┐      ┌──────────────────────────┐
│  后台线程（生产者）        │      │  UI 线程（消费者）          │
│  Task.Run + while(true)  │      │  DispatcherTimer 每 400ms  │
│  每秒 ~200 条数据         │  →   │  批量 TryDequeue + 渲染     │
│  Enqueue 入队            │      │  Add 到 ObservableCollection│
└──────────┬──────────────┘      └────────────┬─────────────┘
           │                                  │
           ▼                                  ▼
    ╔══════════════════════════════════════════════╗
    ║     ConcurrentQueue<DeviceData>              ║
    ║     线程安全的缓冲区（蓄水池）                 ║
    ╚══════════════════════════════════════════════╝
```

**核心代码链路：**

```csharp
// 生产者：MockDataService.Start()
Task.Run(() => {
    while (_isRunning) {
        _dataQueue.Enqueue(data);   // 入队
        Thread.Sleep(500);          // 每 500ms 生产一条
    }
});

// 消费者（限流器）：DeviceViewModel.OnTimerTick()
while (_dataService.DataQueue.TryDequeue(out var data)) {
    DataPoints.Add(data);           // 出队 → UI 数据
    _tempPoints.Add(...);           // 出队 → 图表数据
}
```

**核心设计思想：数据生产速度 ≠ UI 刷新速度。** 生产者随便多快，UI 每 400ms 才批量取一次。

### 4.2 滑动窗口（阶段二）

```csharp
while (DataPoints.Count > 50)   // 超过 50 条就丢掉最旧的
{
    DataPoints.RemoveAt(0);     // RemoveAt(0) 是 O(n)，但 50 条可以接受
    _tempPoints.RemoveAt(0);    // 三个集合同步删除，保持一致性
    _pressPoints.RemoveAt(0);
}
```

**为什么限 50 条？** 不限制的话，跑 1 小时 = 3600×2 = 7200 条，跑一天 = 17 万条。`ObservableCollection` 越来越大，内存和 UI 渲染都会卡死。

### 4.3 DispatcherTimer vs Task.Run（阶段二）

| | DispatcherTimer | Task.Run |
|------|----------------|----------|
| 运行线程 | **UI 线程** | 线程池（后台线程） |
| 用途 | 定时更新界面 | 后台计算/IO |
| 能不能操作 UI？ | ✅ 可以，直接改 `ObservableCollection` | ❌ 不可以，会抛跨线程异常 |
| 这个项目里谁在用？ | DeviceViewModel（消费者） | MockDataService（生产者） |

### 4.5 报警引擎 + 3 秒防抖（阶段 4a）

```
正常          超标发生          持续超标         恢复            解除
────●────────────●────────────────●───────────────●──────────────●──→ 时间
               │←── 报警！──────────→│←─ 3秒观察期 ─→│ 解除报警
                IsAlarming=true      保持报警         IsAlarming=false
```

**防抖逻辑代码：**

```csharp
if (tempError || pressError)
{
    _lastViolationTime = DateTime.Now;      // 每次超标都刷新时间戳
    if (!IsAlarming)                         // 第一次超标：记录起点
    {
        _currentAlarmStartTime = DateTime.Now;
        _currentAlarmMaxTemp = 0;
        IsAlarming = true;
    }
    // 持续追踪报警会话期间的最大值
    if (maxTempInBatch > _currentAlarmMaxTemp) ...
}
else  // 本批次正常
{
    if (IsAlarming)
    {
        var seconds = (DateTime.Now - _lastViolationTime).TotalSeconds;
        if (seconds >= 3.0)   // 连续 3 秒正常 → 解除
        {
            IsAlarming = false;
            SaveCurrentAlarmLogAsync();    // 写入 SQLite
        }
        else                   // 不足 3 秒 → 保持报警 + 倒计时
        {
            AlarmMessage = $"观察中... (剩余 {3.0 - seconds:F1} 秒)";
        }
    }
}
```

**XAML 报警视觉效果：**

| 效果 | 实现方式 |
|------|---------|
| 报警灯绿色→红色 | `DataTrigger` on `IsAlarming` → `Fill="#EF4444"` |
| 报警灯呼吸闪烁 | `Storyboard` + `DoubleAnimation` Opacity 1.0↔0.2 |
| 报警消息条背景 | `DataTrigger` → `Background="#FEE2E2"` |
| 报警消息文字 | `DataTrigger` → `Foreground="#DC2626"` |

### 4.6 SQLite 报警存储（阶段 4b）

**和 ContactBook 的 JSON 存储对比：**

| | JSON 文件（ContactBook） | SQLite（IndustrialMonitor） |
|------|------------------------|---------------------------|
| 包 | 无（`System.Text.Json` 内置） | `Microsoft.Data.Sqlite` |
| 存储格式 | 一个 JSON 文件 | 一个 `.db` 二进制文件 |

```sql
    MaxValue REAL NOT NULL,
    Threshold REAL NOT NULL,
    StartTime TEXT NOT NULL,
    EndTime TEXT NOT NULL

```

**参数化插入（防 SQL 注入）：**

```csharp
command.Parameters.AddWithValue("$deviceId", record.DeviceId);
```

**XAML DataGrid 绑定：**

### 4.7 依赖注入（DI）容器（阶段五）

**改前（ContactBook 方式）：**
```csharp
// ViewModel 自己 new Service —— 硬编码依赖，无法替换
private readonly ContactStorage _storageService = new ContactStorage();
```

**改后（DI 方式）：**
```csharp
// App.xaml.cs：注册
services.AddSingleton<IAlarmLogService, AlarmLogService>();      // 全局唯一
services.AddTransient<IMockDataService, MockDataService>();      // 每次 new
services.AddTransient<DeviceViewModel>();                         // 每个设备一个

// DeviceViewModel：接收接口，不关心谁来实现
public DeviceViewModel(IMockDataService dataService, IAlarmLogService alarmLogService, 
                       IOptionsMonitor<MonitorConfig> configMonitor)
```

**生命周期对比：**

| 生命周期 | 含义 | 本项目的应用 |
|---------|------|------------|
| `Singleton` | 全程序只有 1 个实例 | `IAlarmLogService`（全局共享一个数据库连接） |
| `Transient` | 每次请求都 new | `IMockDataService`（每个设备有自己的数据队列） |

**接口（Interface）的作用：**

```csharp
// 定义契约："能干什么"
public interface IMockDataService
{
    ConcurrentQueue<DeviceData> DataQueue { get; }
    void Start();
    void Stop();
}

// 实现："怎么干"
public class MockDataService : IMockDataService { ... }
```

接口让你可以随时换实现而不改 ViewModel。比如测试时可以注入一个假的 `FakeMockDataService`。

**工厂模式：Func<DeviceViewModel>**

```csharp
// 注册：告诉 DI 如何动态创建设备
services.AddSingleton<Func<DeviceViewModel>>(provider => 
    () => provider.GetRequiredService<DeviceViewModel>());

// 使用：MainViewModel 每次 AddDevice 都通过工厂创建一个新设备
var device = _deviceFactory();
```

`Func<T>` 是一个无参返回 T 的委托。DI 自动注入委托，调用即创建新实例。

### 4.8 配置热重载（阶段六）

**appsettings.json：**
```json
{
  "MonitorConfig": {
    "TemperatureThreshold": 85.0,
    "PressureThreshold": 2.4,
    "RefreshIntervalMs": 400,
    "SlidingWindowSize": 50,
    "AlarmDebounceSeconds": 3.0
  }
}
```

**加载 + 热重载机制：**
```csharp
// App.xaml.cs：构建配置（reloadOnChange: true 启用文件监视）
var builder = new ConfigurationBuilder()
    .SetBasePath(AppDomain.CurrentDomain.BaseDirectory)
    .AddJsonFile("appsettings.json", optional: false, reloadOnChange: true)
    .AddJsonFile(Path.Combine("..", "..", "..", "appsettings.json"), 
                 optional: true, reloadOnChange: true)  // 开发时监视源文件
    .Build();

services.Configure<MonitorConfig>(configuration.GetSection("MonitorConfig"));
```

**DeviceViewModel 里用 IOptionsMonitor<T> 读取：**
```csharp
// 每次访问都拿最新值
public double TemperatureThreshold => _configMonitor.CurrentValue.TemperatureThreshold;

// OnTimerTick 里缓存本次 Tick 的配置快照
var currentConfig = _configMonitor.CurrentValue;

// 订阅文件变化 → 实时更新图表阈值线
_configMonitor.OnChange(OnConfigChanged);

private void OnConfigChanged(MonitorConfig newConfig)
{
    Application.Current?.Dispatcher.Invoke(() =>
    {
        _timer.Interval = TimeSpan.FromMilliseconds(newConfig.RefreshIntervalMs);
        TemperatureSections[0].Yi = newConfig.TemperatureThreshold;
        OnPropertyChanged(nameof(TemperatureThreshold));
    });
}
```

**踩坑记录：** 最初 `services.Configure<T>(IConfigurationSection)` 编译报错，因为缺少 `Microsoft.Extensions.Options.ConfigurationExtensions` 包。后来又发现修改项目源文件不生效，因为 `CopyToOutputDirectory` 只在编译时复制——加入第二条 `AddJsonFile` 指向项目源路径解决。

### 5.6 `Services/MockDataService.cs`（72 行）

**核心算法：90% 正常 + 10% 异常的数据生成**

```csharp
bool isAnomaly = _random.Next(0, 100) < 10;  // 10% 概率异常
if (isAnomaly)
{
    temp = 86.0 + random * 9.0;   // 86~95°C（超过 85°C 阈值）
    press = 2.45 + random * 0.2;  // 2.45~2.65 MPa（超过 2.4 阈值）
}
else
{
    temp = 65.0 + random * 17.0;  // 65~82°C（安全范围）
    press = 1.8 + random * 0.5;   // 1.8~2.3 MPa（安全范围）
}
```

10% 的概率让报警功能可测试，而不是永远正常或永远报警。

```csharp
// 图表配置

public RectangularSection[] TemperatureSections { get; set; }  // 阈值虚线
```

**OnTimerTick 完整流程（每条都重要）：**

```
1. 声明 maxTempInBatch / maxPressInBatch = 0
2. 获取配置快照：var cfg = _configMonitor.CurrentValue
3. while TryDequeue → 出队 → Add 到三个集合 → 更新 max
4. while Count > cfg.SlidingWindowSize → RemoveAt(0)（滑动窗口）
5. DeviceCount = Count（更新计数）
6. 取最新一条 → 更新 LatestTemperature/LatestPressure 文本
7. if 有超标 → 报警状态机（更新防抖时间戳、报警起点检测、追踪最大值、拼消息）
8. else if 报警中 → 检查 3 秒到期 → 解除或保留+倒计时
```

**报警会话最大值的用途：** 一次报警从触发到解除，温度可能从 86→92→88→83。日志记录的是 92（整个会话的峰值），而不是触发时的 86 或解除时的 83。

### 5.11 `App.xaml.cs`（83 行）

**DI 容器的 Composition Root（组合根）——整个应用对象图的唯一来源：**

```csharp
protected override void OnStartup(StartupEventArgs e)
{
    // 1. 加载配置（含热重载）
    var config = new ConfigurationBuilder()
        .AddJsonFile("appsettings.json", optional: false, reloadOnChange: true)
        .AddJsonFile("../../../appsettings.json", optional: true, reloadOnChange: true)
        .Build();

    // 2. 注册所有服务到容器
    var services = new ServiceCollection();
    services.Configure<MonitorConfig>(config.GetSection("MonitorConfig"));
    services.AddSingleton<IAlarmLogService, AlarmLogService>();
    services.AddTransient<IMockDataService, MockDataService>();
    services.AddSingleton<MainViewModel>();
    services.AddTransient<DeviceViewModel>();
    services.AddSingleton<Func<DeviceViewModel>>(p => () => p.GetRequiredService<DeviceViewModel>());
    services.AddSingleton<MainWindow>();

    // 3. 构建容器
    ServiceProvider = services.BuildServiceProvider();

    // 4. 从容器获取 MainWindow（递归解析所有依赖）
    var mainWindow = ServiceProvider.GetRequiredService<MainWindow>();
    mainWindow.Show();
}
```

## 七、你在这个项目中踩过的坑

| 坑 | 原因 | 修复 |
|----|------|------|
| `IOptionsMonitor.OnChange` 不触发 | 修改的是项目源文件而非 bin/Debug 下的副本 | 加第二条 `AddJsonFile` 指向项目源路径 |

| 数据出队循环丢失（两次！） | Gemini 生成的代码漏掉了 `while TryDequeue` | 手动补回 |
| 报警会话起点 `_currentAlarmStartTime` 永远不记录 | `IsAlarming = true` 写在 `if (!IsAlarming)` 前面 | 调换顺序：先判断旧状态，再设新状态 |

| `OptionsBuilder.Bind` 不会热重载 | 只在注册时执行一次 | 改用 `services.Configure<T>(IConfigurationSection)` 重载 |

## 九、面试中可能被问到的问题

### 1. "这个项目你是怎么保证 UI 不卡顿的？"
> 生产者-消费者模式 + DispatcherTimer 限流。后台线程每秒产生数据塞进 ConcurrentQueue，UI 线程每 400ms 批量 TryDequeue 取数据，一次只渲染一批。滑动窗口限制只展示最近 50 个数据点，防止 ObservableCollection 无限增长。

### 2. "报警引擎的防抖是怎么设计的？"
> 每次 Tick 检查本批次最大值是否超阈值。一旦超标立刻报警并记录时间戳。连续 3 秒无超标才解除。解除前的倒计时显示在 UI 上。整个报警会话的峰值（而非触发值或解除值）被记录到 SQLite。

### 3. "依赖注入在这个项目中怎么用的？"
> `App.xaml.cs` 作为 Composition Root，用 `ServiceCollection` 注册了 8 个服务。Singleton 用于全项目共享的 AlarmLogService 和 MainViewModel，Transient 用于每个设备独立的 MockDataService 和 DeviceViewModel。用 `Func<DeviceViewModel>` 工厂委托实现动态创建设备。

### 4. "配置热重载是怎么实现的？"
> `appsettings.json` 通过 `ConfigurationBuilder.AddJsonFile(reloadOnChange: true)` 加载，注册为 `IOptionsMonitor<T>`。`OnChange` 回调里切回 UI 线程更新 `_timer.Interval` 和图表阈值线。同时监视项目源文件和输出目录两份路径，开发时编辑源文件即生效。

### 5. "为什么用 SQLite 而不是 JSON？"
> 报警记录需要按时间倒序查询、按设备筛选。JSON 只能全量读写，SQLite 支持 SQL 查询。而且 SQLite 是嵌入式数据库，不需要安装服务，和数据文件一样便携。
